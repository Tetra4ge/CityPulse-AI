# CityPulse AI — Database Schema

## Is a database "required" in Phase 0?

**Yes, the schema must be defined in Phase 0 — but no live database connection is required yet.** Here's the reasoning:

- Every later phase (ingestion, agents, decision logging, the timeline UI) depends on a stable table shape. If the schema is decided ad hoc per phase, you will spend Phase 3-4 rewriting Phase 1 code.
- Defining the schema now, as SQL DDL files committed to the repo, costs almost nothing and prevents that churn.
- **Actually connecting to live BigQuery is not required until Phase 1**, when the Ingestion Agent needs somewhere real to write. Phase 0 ships the schema as files plus a local-dev equivalent so the API stubs have something to validate their mock responses against, without needing real GCP credentials to run `npm run dev`.

**Recommendation: use a local SQLite (or Postgres via Docker) database for local development**, with the schema written so it maps 1:1 to the BigQuery production schema. This lets every contributor run the full stack without GCP credentials, and the swap to real BigQuery in Phase 1 is a connection-string change, not a schema rewrite.

## Tables

These map directly to `03_TRD.md` Section 4. Below is the full DDL, written in BigQuery Standard SQL, with notes on the local-dev SQLite equivalent.

### `aqi_history`

```sql
CREATE TABLE IF NOT EXISTS aqi_history (
  id STRING DEFAULT GENERATE_UUID(),
  zone STRING NOT NULL,
  timestamp TIMESTAMP NOT NULL,
  aqi_value FLOAT64 NOT NULL,
  source_status STRING NOT NULL,        -- 'ok' | 'stale' | 'failed'
  confidence_penalty FLOAT64 DEFAULT 0.0,
  raw_payload_uri STRING,               -- pointer to Cloud Storage raw object
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP()
)
PARTITION BY DATE(timestamp)
CLUSTER BY zone;
```

### `weather_history`

```sql
CREATE TABLE IF NOT EXISTS weather_history (
  id STRING DEFAULT GENERATE_UUID(),
  zone STRING NOT NULL,
  timestamp TIMESTAMP NOT NULL,
  temperature_c FLOAT64,
  humidity_pct FLOAT64,
  wind_kph FLOAT64,
  source_status STRING NOT NULL,        -- 'ok' | 'stale' | 'failed'
  raw_payload_uri STRING,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP()
)
PARTITION BY DATE(timestamp)
CLUSTER BY zone;
```

### `complaints_history`

```sql
CREATE TABLE IF NOT EXISTS complaints_history (
  id STRING DEFAULT GENERATE_UUID(),
  zone STRING NOT NULL,
  timestamp TIMESTAMP NOT NULL,
  raw_text STRING NOT NULL,
  category STRING,                      -- e.g. 'respiratory', 'visibility', 'odor'
  severity STRING,                      -- 'low' | 'medium' | 'high' | 'very_high'
  cluster_id STRING,                    -- set by Triage Agent's cuML clustering step
  is_simulated BOOL DEFAULT TRUE,       -- explicit flag per the project's data honesty statement
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP()
)
PARTITION BY DATE(timestamp)
CLUSTER BY zone;
```

### `forecasts`

Not in the original TRD table list but needed to persist Forecast Agent output for the timeline and for what-if comparisons — added here for completeness.

```sql
CREATE TABLE IF NOT EXISTS forecasts (
  id STRING DEFAULT GENERATE_UUID(),
  zone STRING NOT NULL,
  generated_at TIMESTAMP NOT NULL,
  predicted_aqi FLOAT64 NOT NULL,
  horizon_hours INT64 NOT NULL,
  confidence FLOAT64 NOT NULL,
  reasoning STRING,
  is_whatif BOOL DEFAULT FALSE,         -- true if generated by the what-if simulator, not a scheduled run
  whatif_params JSON,                   -- e.g. { "traffic_multiplier": 1.2 }
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP()
)
PARTITION BY DATE(generated_at)
CLUSTER BY zone;
```

### `agent_decisions_log`

The most important table — backs the audit trail, the conflict/escalation record, and the Agent Activity Timeline UI.

```sql
CREATE TABLE IF NOT EXISTS agent_decisions_log (
  id STRING DEFAULT GENERATE_UUID(),
  agent_name STRING NOT NULL,           -- 'ingestion' | 'forecast' | 'triage' | 'decision' | 'reflection' | 'human' | 'notification'
  zone STRING,
  timestamp TIMESTAMP NOT NULL,
  action STRING NOT NULL,               -- short human-readable action, e.g. "Detected complaint spike, escalating"
  input_ref STRING,                     -- id/reference to the input this agent acted on
  output_json JSON,                     -- full structured output for this agent step
  conflict_flag BOOL DEFAULT FALSE,
  escalation_flag BOOL DEFAULT FALSE,
  confidence FLOAT64,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP()
)
PARTITION BY DATE(timestamp)
CLUSTER BY agent_name, zone;
```

### `decisions`

Persists the Decision Agent's actual recommendation objects (separate from the log, which records every agent step — this table is specifically the recommendation records that flow into approval/notification).

```sql
CREATE TABLE IF NOT EXISTS decisions (
  id STRING DEFAULT GENERATE_UUID(),
  zone STRING NOT NULL,
  generated_at TIMESTAMP NOT NULL,
  risk_level STRING NOT NULL,           -- 'low' | 'medium' | 'high' | 'severe'
  overall_confidence FLOAT64 NOT NULL,
  conflict_detected BOOL DEFAULT FALSE,
  recommendations JSON NOT NULL,        -- array of { target, action }
  rationale STRING,
  reflection_id STRING,                 -- FK-equivalent reference to reflections.id
  approval_status STRING DEFAULT 'pending', -- 'pending' | 'approved' | 'rejected'
  reviewer_id STRING,
  reviewed_at TIMESTAMP,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP()
)
PARTITION BY DATE(generated_at)
CLUSTER BY zone;
```

### `reflections`

```sql
CREATE TABLE IF NOT EXISTS reflections (
  id STRING DEFAULT GENERATE_UUID(),
  decision_id STRING NOT NULL,
  validated BOOL NOT NULL,
  requires_human_review BOOL NOT NULL,
  flags JSON,                           -- array of strings, e.g. ["low_confidence", "stale_weather_data"]
  notes STRING,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP()
)
CLUSTER BY decision_id;
```

### `notifications_log`

```sql
CREATE TABLE IF NOT EXISTS notifications_log (
  id STRING DEFAULT GENERATE_UUID(),
  decision_id STRING NOT NULL,
  zone STRING NOT NULL,
  timestamp TIMESTAMP NOT NULL,
  target_audience STRING NOT NULL,      -- 'schools' | 'hospital' | 'transit' | 'citizens'
  message STRING NOT NULL,
  approved_by STRING,
  dispatch_status STRING DEFAULT 'simulated', -- 'simulated' | 'sent' | 'failed'
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP()
)
PARTITION BY DATE(timestamp)
CLUSTER BY zone;
```

## Local development database

For Phase 0-1 local development without GCP credentials, mirror the above as SQLite (via Prisma or Drizzle ORM, recommended for type-safe access from the Next.js API routes):

- `JSON` columns → SQLite `TEXT` storing serialized JSON (Prisma/Drizzle both handle this transparently)
- `TIMESTAMP` → SQLite `DATETIME` / ISO string
- `GENERATE_UUID()` → application-layer UUID generation (`crypto.randomUUID()`)
- Partitioning/clustering → no equivalent needed locally, BigQuery-specific performance feature, ignored in dev

**Recommended Phase 0 task:** set up Drizzle ORM (or Prisma) with a schema file that generates both the local SQLite migration and stays close enough to the BigQuery DDL above that translating to a real BigQuery client in Phase 1 is mechanical, not a redesign.

## What Phase 0 actually needs to deliver for the DB

1. All DDL files above saved under `src/lib/db/schema/*.sql` (BigQuery versions, committed as the source of truth for production)
2. A Drizzle/Prisma schema file generating the local SQLite equivalent
3. A seed script (`scripts/seed.ts`) that populates a handful of plausible rows per table, so the Phase 0 placeholder UI and API stubs have something real-looking to return instead of empty arrays
4. `bigquery-client.ts` exists as a typed stub (function signatures matching what Phase 1 will need) but does not actually connect — it should throw a clear "not implemented until Phase 1" error if called, so it's obvious if something accidentally tries to use it early
